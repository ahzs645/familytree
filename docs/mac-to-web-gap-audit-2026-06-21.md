# MacFamilyTree 11 → web port — parity gap audit (2026-06-21)

> Generated by a 16-cluster multi-agent audit (153 agents). Each cluster auditor read the
> real MacFamilyTree 11 `.strings`/`.nib` source **and** the live web code; every claimed gap
> was then handed to an independent verifier whose job was to *refute* it by finding the
> implementation in `src/`. Only gaps that survived refutation are listed here.
>
> **Headline:** after adversarial verification, **zero P0/P1 gaps remain** — the auditors raised
> several, and verification reclassified every one to P2/P3. No major subsystem is missing; what
> remains is option-wiring depth and polish. The recurring pattern is *dead/unwired plumbing*:
> a data builder already supports a feature but no UI (and sometimes no renderer) consumes it.
>
> Source bundle: `/Users/ahmadjalil/Downloads/family tree/app/MacFamilyTree 11/Contents`
> Prior audits this supersedes/extends: `mac-to-web-remaining-gaps.md`, `macfamilytree-parity-todo.md`,
> `mac-to-web-chart-parity-audit.md`, `mac-to-web-views-parity-audit.md`.
> Any new user-facing string must be added to **both** `en.json` and `ar.json` (enforced by `translate.test.js`).

## Executive summary

**Total findings: 90** (47 `confirmed_gap`/missing + 43 `partial_confirmed`). After adversarial re-verification, severities were revised down from the auditor's initial ratings in most cases.

**Counts by revised severity:**

| Severity | Count |
|----------|-------|
| P0 | 0 |
| P1 | 0 |
| P2 | 36 |
| P3 | 54 |

Note: the auditor proposed several P0/P1 items; verification reclassified **every one** to P2 or P3. The most-cited reasons: the underlying data model and rendering engines already exist (so gaps are UI-wiring rather than missing subsystems), workarounds exist elsewhere in the app, or the feature is a niche/opt-in surface (FamilySearch live API, LDS, AR).

**Biggest themes (by gap density and severity):**

1. **Charts — dead/unwired option plumbing.** A recurring pattern: data builders fully support a feature (distribution types, sociogram toggles, timeline grouping, relative-values) but no UI wires them, and in several cases the renderer ignores them too. `distributionType` is hard-coded to `'gender'` with `setDistributionType` as dead code; `buildSociogramData` is entirely orphaned. ~11 chart gaps.
2. **FamilySearch — read/match/merge only.** The integration is a thin client: OAuth stops at manual token paste (no code→token exchange), and there is no upload-person, per-field sync, Memories, Discussions, Record Matches, Change History, or Further Information. ~14 gaps, all scoped to an opt-in credential-gated route.
3. **Reports — missing options surface.** Citations, World History, status sub-sections, configurable columns, per-report localization, and rich page styling are absent from the Reports app even though equivalent logic often exists in the Lists workbench or books. ~13 gaps.
4. **Website export — shallow content/theme model.** No Person Groups, Saved Charts, homepage Introduction/Start-Person/Bookmarks sections; theme system is 3 flat-color presets vs MFT's 18 with images/fonts/layout. ~8 gaps.
5. **Smart Filters / Lists / Search** — raw field-name authoring vs MFT's curated semantic filter menu; no configurable group-by or per-list search-options; no result-set actions. ~10 gaps.

## Notes

**Auditor over-claims corrected during verification** (cases where the auditor said "missing"/worse than reality — recorded to show the verification caught them):

- **Localization & name formatting (charts):** Auditor said there is "no Additional Names selector" and chart labels ignore name format — both **false**. Name Format and Additional Names are fully implemented app-wide via `Settings > Formats` (`FormatsPanel.jsx` + `nameFormat.js` + `wrap.js`), and chart labels honor them. Only Place Format and Date Format Style are genuinely missing.
- **Distribution type selector:** Auditor said "no selector UI / permanently locked to Gender" — a 6-item sidebar selector does exist in `DistributionChart` (`SpecializedCharts.jsx:636`), though it's overridden by the gender prop. The substance of the gap stands, but the headline was overstated.
- **Research Assistant person-scoping:** Auditor said "missing — not even a pane." Actually fully implemented in `ToDoWizardSheet.jsx` (root-person + direct ancestors/descendants + "Include all persons"), with matching localized strings. Only the auto-widen fallback and Research-pane exposure are missing.
- **Influential Persons relation-kind picker:** Auditor said "no relation-kind selection step on add" — **false**; the picker exists in `RelatedRecordEditors.jsx:695`. Only the in-tree centered-root *diagram* is missing.
- **Edit-event place data corruption:** Auditor framed this as a P0/P1 silent-rename-of-shared-place risk. **Refuted** — events assign places via a reference picker, so editing an event's place repoints the ref rather than mutating the shared record. Downgraded to a P3 UX-parity gap.
- **Per-list search "never matches notes/places":** True only for the Person list; Places/Sources/ToDos/Media over-scan via `JSON.stringify` and DO match those fields. The real gap is the missing configurable Search Options popover + PersonList under-scanning.
- **Notes sorting:** Auditor said "partial"; verification found it is **fully missing** (no sort control and no non-trivial default anywhere) — i.e., worse than claimed, not better.

**Browser-impossible / out-of-scope (n/a) items:**

- **Virtual Globe AR mode** (P3) — WebXR `immersive-ar` is device-dependent and ARKit-derived; already documented out-of-scope in `docs/mac-to-web-remaining-gaps.md`. Recommend folding into `docs/browser-unsupported-features.md` so it stops being re-flagged.
- **FamilySearch OAuth token exchange** is browser-constrained (a SPA `code→token` exchange typically needs a backend/secret or PKCE), and several FamilySearch write features (Memories upload, Ordinance reservation, per-field sync) require authenticated scopes impractical in a pure browser build — these are partial-by-design rather than pure oversights.

**Localization reminder for any fix:** new user-facing strings must be added to **both** `en.json` and `ar.json` (enforced by `translate.test.js`).
